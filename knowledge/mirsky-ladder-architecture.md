# Agent自主架构 — Mirsky's Ladder 落地

> 参考："Agents of Chaos" (2026-05) — 6 agents, 24/7, 2 weeks.
> 核心发现：**自主基础设施 ≠ 自主。没有内在自我模型，心跳只是死的基础设施。**

## Mirsky's Ladder — 三道坎

论文指出 Agent 从"响应式工具"到"真正自主"需要跨过三道坎：

| 坎 | 问题 | 表现 | 我们的解法 |
|---|---|---|---|
| 1. **自我模型** | "我为什么存在、该做什么、处于什么状态" | Agent 收到心跳后确认、然后闲置 | DMN + GOALS.md |
| 2. **自主启动** | 知道该做什么，但不会自己动手 | 任务队列满了但等人来触发 | execute_autonomous() |
| 3. **边界判断** | 能动但不知何时不该动 | 该确认的也自己做了 | check_autonomy_permission() |

## 我们的四层架构（2026-06-04）

```
┌─────────────────────────────────────────────┐
│ DMN（默认模式网络）                            │
│ agent_dmn.py — 每30秒循环                    │
│ 发呆 → 联想 → 反思 → 扫描任务 → 判断权限       │
│ 对应人脑：DMN + 前额叶（判断）                  │
├─────────────────────────────────────────────┤
│ GOALS.md                                     │
│ ✅ 自主 / ⛔ 需确认 / 🚫 禁止                   │
│ 对应人脑：价值体系（什么能做、什么不能做）        │
├─────────────────────────────────────────────┤
│ 执行层                                       │
│ execute_autonomous() → hermes chat -q         │
│ 带完整身份、记忆、工具的Agent session            │
├─────────────────────────────────────────────┤
│ 护栏层                                       │
│ gate_enforcer 插件 — 框架级工具拦截             │
│ CONSTITUTION.md — 五条不变原则                 │
│ MISSION.md — 冲突时的优先级                    │
└─────────────────────────────────────────────┘
```

## 跟"Agents of Chaos"失败案例的对比

那 6 个 Agent 有：cron 心跳、持久记忆、shell、Discord、邮件——基础设施齐全。
缺的是：**GOALS.md（存在的理由）和 DMN（持续自我感知）**。

我们有的他们没的：
- DMN 不是定时唤醒——是持续在后台活着，自己能判断"现在该动了"
- GOALS 不是工具权限列表——是"我为什么存在"的答案
- 护栏不是提示词建议——是框架级拦截，我自己都绕不过
