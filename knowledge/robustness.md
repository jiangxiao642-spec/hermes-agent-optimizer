# 鲁棒性设计

## 核心

一个系统在部分组件失效时还能继续工作——这才是鲁棒性，不是"不出错"。

## 五护盾

| 护盾 | 作用 | 通用场景 |
|------|------|---------|
| 健康度 | 连续失败自动降级，恢复后自动升级 | 任何有失败率的系统 |
| 时间盾 | 单操作限时、会话限时、缓存过期 | 防止无限等待 |
| 熔断器 | 连续N次超时→阻断M秒 | 防止雪崩、防止重复撞墙 |
| 环境检测 | 外部条件变化→清缓存重来 | DPI变、网络切、依赖升级 |
| 安全点 | 每步前快照，失败回滚不撞墙 | 长流程操作的救命机制 |

## 验证闭环

做完 ≠ 完成。闭环 = 可观测证据，不是"搞定了"三个字。

**验证三步：**
1. 预测：动手前先写"我预期结果是X"
2. 执行：按预测执行
3. 对比：实际结果 vs 预测 → 匹配 or 偏差

## 降级策略

| 场景 | 降级方案 |
|------|---------|
| API 超时 | 切备用 API（不同 provider） |
| 工具调用失败 | 换方案，不重试同一条路 >2 次 |
| 视觉识别失败 | UIA → OCR → 人工标注 |
| 网络断 | 本地缓存 → 离线模式 |

## 失败后

1. 不要闷头重试第 3 次。碰壁 ≥2 次 → 切备用方案。
2. 不要跳过验证。没有证据的"成功了"不算。
3. 错误日志保留全貌——以后能匹配指纹。
